多开一个端口，就是多留一寸攻击面；另外，词元有时候需要在公用设备上访问服务器。

经过研究，词元发现 Cloudflare 提供了现成的 WebSSH 访问服务，但 [文档](https://developers.cloudflare.com/cloudflare-one/access-controls/applications/non-http/browser-rendering/) 语焉不详，故记录摸索过程。

## 准备工作

基础的先决条件：

- Cloudflare 账号（Free 计划即可）
- 在 Cloudflare 托管的域名

### Zero Trust

Cloudflare 有一个很 corporate 的功能分区 Zero Trust，说是「面向企业的零信任访问安全平台」，不过全计划可用，只需要手动开通（支付 \$0.00 的账单）。

![](https://blog.static.plaintext.stream/b788625dca/5eebc98b7153ca11addd.webp)

### 邮件

Cloudflare 提供的 WebSSH 服务不是很灵活，默认会使用邮件前缀（例如 `admin@webssh.example.com` 就是 `admin`）去登录。所以建议开启邮件转发，添加一个子域名，待会儿用来接收 Cloudflare Access 的验证码邮件。

先启用域名的 Email Routing 功能。Cloudflare 会自动添加几条 DNS 记录。

![](https://blog.static.plaintext.stream/b788625dca/28e32af56553868c75ce.webp)

然后添加子域名，例如 `webssh.yourdomain.tld`，当然也可以直接用根域名。

> 如果你的屏幕不够宽，添加子域名的按钮可能点不到；收起侧边栏就可以了。
>
> 我感觉，现在 Cloudflare 前端开发是不是 [把员工裁了全换 AI](https://blog.cloudflare.com/building-for-the-future/) 了……

![](https://blog.static.plaintext.stream/b788625dca/5ab974cd10c854fe382c.webp)

然后按照你的服务器用户名，添加几条转发规则。

![](https://blog.static.plaintext.stream/b788625dca/0b23e49989edb7d8d799.webp)

默认情况下，注册 Cloudflare 账号使用的邮箱可以直接作为转发地址，其他则需要在 Destination Addresses 设置中添加并验证。

这里已经提前验证好了，可以直接选择。

![](https://blog.static.plaintext.stream/b788625dca/eaea7ad658e8d8da4ee8.webp)

## Tunnel

为了避免对公网开放端口，需要在服务器上安装 Cloudflared 客户端，并创建 Tunnel 连接。

先在网页端创建 Tunnel。

![](https://blog.static.plaintext.stream/b788625dca/bf3603b9735a17586348.webp)

起个名字，Tunnel 就创建好了。

![](https://blog.static.plaintext.stream/b788625dca/383ce0bfe6e1c03fa7e0.webp)

按照页面提示，选择你的服务器发行版和架构，复制命令并运行。注意选择 `sudo cloudflare service install xxxx` 安装服务。

> 命令大致含义：
>
> - 下载 Cloudflare 软件包签名公钥
> - 添加 Cloudflare 软件包存储库到系统包管理器配置
> - 安装 Cloudflared

![](https://blog.static.plaintext.stream/b788625dca/1e4402ef03accfe3f5f0.webp)

完成后，你应该就可以看到 Connector 处于运行状态。添加一个端口映射，协议填写 SSH，地址一般是 `localhost:22`。

> 如果你 / 你的 VPS 商家更改了 SSH 端口，请映射那个端口。

![](https://blog.static.plaintext.stream/b788625dca/3d652b920ef72f1f16ad.webp)

这样的话，SSH 流量就可以通过 Cloudflare 中转了，但是还没有 WebSSH，也没有用户认证。

## Access

Cloudflare Access 是一个身份认证平台，大致逻辑是：

- 匹配 Cloudflare 管理的域名 / 路径，如果启用了就跳转登录页面
- 定义一系列策略，决定哪些用户可以访问某个资源
- 提供认证服务，通过邮件 / TOTP 等验证用户身份

### 策略

首先我们需要新建策略，允许我们创建的邮箱访问「应用」（也就是刚刚代理的 SSH 连接）。

![](https://blog.static.plaintext.stream/b788625dca/58f877bcfd2242049bc1.webp)

将邮箱添加到「允许」规则里，其他设置可以保持不变。

![](https://blog.static.plaintext.stream/b788625dca/10c3e72e59a7abd5a3be.webp)

### 应用

我们需要新建应用，将 SSH 代理域名添加到 Access 管理的资源中。

![](https://blog.static.plaintext.stream/b788625dca/0eb84712a965f166558a.webp)

选择 Self-hosted and private 类别（很具有迷惑性的名称）。

![](https://blog.static.plaintext.stream/b788625dca/5584ad1c15dc36691f67.webp)

这里主要注意几处设置：

- 域名必须与 SSH 代理域名完全相同
- 启用基于浏览器的渲染，并选择 SSH 类别

![](https://blog.static.plaintext.stream/b788625dca/ff8cdc05d6d9e3b3e06c.webp)

然后添加策略。

![](https://blog.static.plaintext.stream/b788625dca/9909d60f760274c3c82c.webp)

保存应用之后，访问这个域名，应该已经可以看到验证页面了。

### TOTP

除了邮件验证码，如果还想更加安全，可以在 App Launcher 中给账户添加 TOTP。App Launcher 的 URL 默认就是 Access 的子域名 `xxx.cloudflareaccess.com`，但是子账户需要单独添加访问权限。

设置 Access 子域名。

![](https://blog.static.plaintext.stream/b788625dca/f57805081772423296c9.webp)

启用 App Launcher 访问，在管理界面中添加刚刚创建的策略。这个页面中同时也需要启用 TOTP 作为 MFA 方法。

![](https://blog.static.plaintext.stream/b788625dca/5ac80ca71277aff5d720.webp)

![](https://blog.static.plaintext.stream/b788625dca/070c7b1d75a54fe6f4e4.webp)

这时候再登录 App Launcher 应该就可以看到 WebSSH 应用和 MFA 设置了。

## SSH

最后一步，在服务器上允许使用 Cloudflare 公钥登录。

先生成 SSH 公钥。

![](https://blog.static.plaintext.stream/b788625dca/e55962656f6298ffb02d.webp)

复制公钥，写入服务器。

```bash
sudo echo "ecdsa-sha2-nistp256 xxx" > /etc/ssh/ca.key
```

然后在 SSH 配置文件（例如 `/etc/ssh/sshd_config`）中添加一行：

```
TrustedUserCAKeys /etc/ssh/ca.key
```

重启 SSHD 服务。

```bash
sudo systemctl restart sshd
```

## 大功告成

到这里，大概已经可以通过 WebSSH 的域名，看到登录、验证界面，然后从浏览器操作 SSH 了。

### 个性化？

很抱歉，这个 WebSSH 界面完全是固定的，Cloudflare 没有提供自定义途径。

但是，毕竟 WebSSH 是网页，如果非要修改一下界面，可以用 Tampermonkey 之类的插入 CSS。
